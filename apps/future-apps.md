# Future Applications

Candidate “Check” applications for the AI in Action educational portfolio. **None of these are committed delivery dates.** Admit a new app only when capacity exists and Phase 1–style validation for current apps is healthy.

## Portfolio sketch

```
AI in Action
│
├── RetireCheck                         ← live
├── SleepCheck                          ← live
├── AI Production Readiness Advisor     ← App #3 engineering showcase · live
├── HabitCheck                          ← App #4 · Better Living · discovery complete
├── FitnessCheck                        ← backlog
├── NutritionCheck                      ← backlog
├── MindCheck                           ← backlog
├── HealthCheck                         ← backlog (extra regulatory caution)
└── Other domain apps
```

- App #3 engineering showcase: [ai-production-readiness-advisor.md](./ai-production-readiness-advisor.md)  
- App #4 Better Living (next build): [MVP spec](../docs/discovery/habitcheck-02-mvp-specification.md) · [habitcheck.md](./habitcheck.md) · [architecture](./habitcheck-architecture.md)

## Candidates

| App | Problem space | Caution |
|-----|---------------|---------|
| **AI Production Readiness Advisor** | Guided production-readiness assessment for AI features | Advisory only — not certification/audit · **live** |
| **HabitCheck** | Weekly targets, kind recovery, first-class AI coach OS (v1.0) | Local-first · **P0 scaffold** · [repo](https://github.com/weidong808/HabitCheck) |
| **FitnessCheck** | Habit / activity tracking with practical UX | Avoid medical claims |
| **NutritionCheck** | Meal / habit support | Avoid diet-medical claims |
| **MindCheck** | Focus / stress / journaling-style support | Wellness framing; mental-health claims carefully |
| **HealthCheck** | General health organization | Highest regulatory risk — do not start casually |

## Admission criteria

Before starting implementation:

1. Clear user problem and narrow v1  
2. Engineering lesson worth a public case study  
3. v1 possible without regulated medical device claims  
4. Independent ship path (shared foundations optional)  
5. Owner bandwidth after polish/content obligations on live apps  

## Shared-foundations expectation

Future apps should eventually plug into shared identity patterns, capability policies, and design conventions — but **ship first**, share later, following Build → Validate → Improve → Document → Share.
